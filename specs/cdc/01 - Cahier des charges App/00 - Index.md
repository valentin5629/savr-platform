## Décisions structurantes prises

### Architecture & Stack
- **Architecture 2 apps** : Plateforme Savr + Savr TMS, communicants via API. Permet d'isoler les risques et les cycles de dev
- **Stack V1 tranchée** : Supabase (DB + Auth + Storage + RLS + Edge Functions) + Resend (emails transactionnels, ~20$/mois) + Puppeteer self-hosted Railway (génération PDF, ~10$/mois) + event-driven webhooks + retry policy uniforme
- **Sous-domaines** : `app.gosavr.io` (Plateforme), `tms.gosavr.io` (TMS), `api.gosavr.io/webhooks/*` (webhooks entrants)
- **RLS multi-tenant** : filtrage par `organisation_id`, `lieu_id`, `created_by`, `client_organisateur_organisation_id` selon le profil
- **Identité visuelle** : refonte complète V1 en gardant couleurs historiques Savr (inspirations à fournir par Val)
- **Intégration MTS-1 conservée en V1** *(ajout 2026-05-29 ; flux + URLs réconciliés sur le relevé as-built 2026-06-06, §3bis §08)* : MTS-1 (produit **MyTroopers**) reste le système de dispatch terrain des transporteurs `type_tms = 'mts1'` (Strike, Marathon) via son **API V3** (host data `*.mytroopers.io/v3`, auth `gateway.*.mytroopers.com/v2` ; doc/console `mts-1.com` à confirmer). La Plateforme **crée la commande + crée la tournée + la dispatch + la valide** (`POST /v3/customerOrders` → `POST /v3/tours` DRAFT → `POST /v3/tours/{tourId}/dispatch` → `PUT validate`) et reçoit les statuts par **POLLING** V3 (`GET /v3/customerOrders` + `GET /v3/tours`, cadence 15-30 min — décision Val 2026-06-05, pas de webhooks V1). Coupure planifiée V2 (cutover TMS Savr). Annule l'ancienne décision « MTS-1 retiré » (qui datait la coupure à tort en V1). Nouveaux champs : `transporteurs.code_transporteur_mts1` + règle `R_code_mts1_requis`. Q3bis-1/2/3 fermées, Q3bis-4 partielle (rate limits + rotation token à confirmer).
