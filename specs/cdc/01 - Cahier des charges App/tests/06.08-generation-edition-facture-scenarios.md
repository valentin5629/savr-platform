|---|----------|-------------|

| F2 | **Mode mensuel = agrégation automatique au batch J+1** (§06.08 fait foi) : la collecte clôturée est ajoutée au brouillon mensuel en cours, brouillon créé si premier de la période. §05 §5 « action manuelle Admin » corrigé. | §05 |
| F3 | **BLOQUANT — `factures_collectes` devient la table des lignes de facture** : `collecte_id` nullable + `CHECK (collecte_id IS NOT NULL OR designation IS NOT NULL)`, colonnes ajoutées `designation text`, `quantite numeric NOT NULL DEFAULT 1`, `taux_tva numeric NOT NULL DEFAULT 20.0`. | F4 | **Numéro fiscal conservé après rejet Pennylane 4xx** : le retour en `brouillon` ne libère pas `numero_facture` ; la re-validation réutilise le même numéro (séquence sans trou). Nouvelle table `sequences_facturation` (`serie`, `annee`, `dernier_numero`, UNIQUE(serie, annee)) — attribution sous verrou ligne (`SELECT ... FOR UPDATE`). | §04, §06.08 |
| F5 | **BLOQUANT — colonne fantôme `factures.marge_logistique` résolue** : colonne ajoutée §04 (decimal, nullable, grain facture = `montant_ht − Σ cout_reparti_ht` des collectes liées), écrite par le trigger cross-schema `fn_recalc_marge_tournee` (conservé, zéro impact tests TMS M04/M07). **Masquage clients** : les rôles clients lisent la vue whitelist `v_factures_client` (toutes colonnes SAUF `marge_logistique`, `erreur_synchro*`) ; SELECT direct table = staff only. | §04, §09, §11 |
| Reco A | **Échéance par défaut = `date_emission + entites_facturation.conditions_paiement_jours`** (plus 30j fixe ; 30 reste le défaut de la colonne). | §06.08 |
| Reco B | **Garde anti-double-facturation** : trigger `trg_fc_collecte_non_facturee` BEFORE INSERT sur `factures_collectes` → rejet si la collecte a déjà une ligne sur une facture `statut ≠ annulee AND type ≠ avoir`. Ce prédicat = définition formelle « collecte non facturée » (sélecteur §6, batch J+1, recyclage post-avoir). | §04, §06.08 |

**Dettes doc corrigées d'office** : (a) §05 liste des séries complétée avec `FPK-YYYY-NNNNN` ; (b) §05 §5 ZD étape 4 réécrite — la validation passe par `en_attente_pennylane`, `emise` uniquement après succès du push (flux unique §06.08 §2 fait foi).
