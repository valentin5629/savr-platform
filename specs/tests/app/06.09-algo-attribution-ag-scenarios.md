## ⚠️ Floues tranchées (décisions prises avant génération)

**F0 — BLOQUANT : table `config_auto_accept_ag` ajoutée §04**
§06.09 §6 décrit une configuration auto-accept (association × type_evenement, toggle actif) mais aucune table n'existe dans §04.

- **Décision (reco tranchée — à confirmer Val)** : création table `config_auto_accept_ag(id uuid PK, association_id uuid FK→associations NOT NULL, type_evenement_id uuid FK→types_evenements NOT NULL, actif bool DEFAULT false, modifie_par uuid FK→users, modifie_le timestamptz, UNIQUE(association_id, type_evenement_id))`.
- RLS : SELECT/UPDATE `admin_savr` uniquement (cohérent `parametres_algo` A9).
- **Propagation : §04 + §09 à compléter avant dev.**

**F1 — TRANCHÉE 2026-06-07 (Val) : saisie `poids_repas_kg` en V1 = Ops Savr manuellement**
En V1 sans TMS Savr, ni Everest ni MTS-1 ne retournent le poids AG. Ops Savr saisit manuellement le poids en kg dans l'onglet "Attributions validées", en analysant les **photos de pesées** remontées par le chauffeur.

- Interface : champ `poids_repas_kg` éditable par Ops Savr + Admin Savr sur la fiche attribution (onglet "Attributions validées")
- Calcul automatique `volume_repas_realise = ceil(poids_repas_kg / parametres_algo.poids_par_repas_kg)` déclenché par un trigger DB sur UPDATE de `poids_repas_kg`
- Le scénario `pesee_tms_webhook_calcule_volume_repas_auto` (cat.6) est mis à jour en conséquence : pas de webhook TMS en V1.
- **Propagation : §06.09 §5 à mettre à jour (retirer "TMS Savr" → "Ops Savr") + §09 ajouter permission UPDATE ops_savr sur `attributions_antgaspi.poids_repas_kg`.**

**F2 — TRANCHÉE 2026-06-07 (Val) : refus association = override standard par l'Admin**
Pas de bouton "L'association a refusé" dédié. Quand une association refuse (par téléphone), l'Admin **rouvre l'attribution depuis l'onglet "Attributions validées"** et applique un override standard (sélectionne Top 2/Top 3 ou recherche libre) avec motif obligatoire (preset `assoc_top1_surchargee` ou `autre`). Pas de workflow automatisé, pas de table de refus historique V1.

- Pas de scénario dédié "refus association" : couvert par les scénarios override existants (cat.1 `override_association_avec_motif_preset`, cat.3 `override_sans_motif_bloque`).
- **Propagation : 0 — §06.09 §2/§3 couvrent déjà ce comportement via l'override.**
